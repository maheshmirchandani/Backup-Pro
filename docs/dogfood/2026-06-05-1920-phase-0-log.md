### 2026-06-05 1930 - PHASE 0 GATE BLOCKER (concerning)

**Surface:** runner / embedded rsync extraction.

**Finding:** First real backup of `~/Documents/1-Imp Docs on iCloud` (1.3 GB, copy mode) completed in 3 seconds, transferred zero bytes, reported `exit status: partial` (exit code 1). Run dir: `/Volumes/ROCKET-2TB/.flashbackup/runs/2026-06-05T1530Z-3042/`. Dest namespace dir `M-Factor-MacBook-local-maheshm/` exists but is empty.

**Root cause:** the rsync embedded into the v0.1.0-core binary is the development placeholder shell script at `internal/rsync/bin/rsync.placeholder`. The script prints "PLACEHOLDER rsync; awaiting Task 12a build" and exits 0. The runner sees exit 0 from T1 transfer, so T1 reports OK. T2 hash-compare finds zero dest files matching the enumerated source list and tags every file `not_transferred`, which collapses into final exit status `partial`.

**Why CI was green:** every e2e test sets `FLASHBACKUP_RSYNC_PATH_FOR_TEST=/opt/homebrew/bin/rsync` (see `cmd/flashbackup/backup_test.go:453`, `:569`). The placeholder code path is never exercised end-to-end in CI with the engine watching it. The placeholder works as a unit-test fixture for the extraction + SHA256 + chmod 0500 + chflags logic only.

**Implication for the v0.1.0-core tag:** the tag was an engine-completeness signal, not a backup-actually-works signal. A clean-install user has no working backup until Task 12a (embedded GNU rsync 3.4.1 universal2 build pipeline) lands. Task 12a was deferred in Plan 1 per `project_execution_state.md`; the deferral was real and tracked, but the tag named `v0.1.0-core` did not communicate the gap.

**Plain-language statement (per global "outcome over architecture" rule):** v0.1.0-core ships a binary that cannot back up real data on a clean install. It is engine-correct but build-incomplete. Phase 0 dogfood cannot close on this binary as-is; it would either need (a) Task 12a to land, or (b) every dogfood run to set the env override, which validates the engine but not the shipped artifact.

**Recovery:** no data loss. Source files intact (copy mode; would have failed atomic gate in move mode and also blocked source deletion). Dest is empty. Safe to re-init the USB and retry.

### Test pyramid gap exposed

No e2e test exercises the embedded payload end-to-end with no env override. An e2e test that ran the binary against the real placeholder and asserted "T1 transfers >0 bytes" would have failed loudly at v0.1.0-core tag time. Such a test would today be `t.Skip("blocked on Task 12a")` until 12a lands. Worth adding as Task 12b once 12a is on the board.

### 2026-06-05 1942 - status free-space reports 0 B (nit)

**Surface:** `flashbackup status` (plain + JSON).

**Finding:** `Free / total: 0 B / 2048.4 GB` (plain) and `"usb_bytes_free": 0` (JSON) on a USB with ~1.3 GB used of 2 TB. The value should be ~2 TB minus used.

**Hypothesis:** APFS volumes on macOS return 0 from `statfs.f_bfree`; the userspace-correct field is `f_bavail`. flashbackup's free-space derivation likely picks the wrong field. Easy fix.

**Severity:** nit. Doesn't affect backup correctness; misleads operators about disk runway.

### 2026-06-05 1942 - verify renderer details path is the generic fallback (nit)

**Surface:** `flashbackup verify` plain renderer summary block.

**Finding:** Output ends with `details: see the <USB>/.flashbackup/ directory` (the renderer's generic-fallback placeholder) instead of pointing at the specific `summary.json` for the verify run (e.g. `<USB>/.flashbackup/runs/2026-06-05T1541Z-2474/verifications/2026-06-05T1542Z-d0c6/summary.json`).

**Context:** Task 38 review I1 specifically reset `UIEvent.Path` semantics so verify ends its summary with the exact artifact file path (`commit 1c682af`). For All-mode aggregates the generic fallback is correct, but default-mode verify (latest run only) should emit the specific summary.json. The runner-side `UIEvent` for the verify-completion event is probably being emitted with `Path` empty, triggering the fallback.

**Severity:** nit. Operator can still find the summary.json by walking `runs/<latest>/verifications/`.

### 2026-06-05 1942 - verify summary duration_seconds integer-truncates to 0 (nit)

**Surface:** `summary.json` for verify.

**Finding:** `"duration_seconds": 0` for a sub-second verify run. The schema should either be a float (`duration_seconds: 0.9`) or an integer milliseconds field (`duration_ms: 943`). Currently a true 0 is indistinguishable from a sub-second run.

**Severity:** nit. Doesn't affect verify correctness; loses precision for sub-second runs which are common with small profiles or fast USBs.

## Sev1 / Sev2 incidents

>
> **Sev2:** verify failure, hash mismatch on a known-good USB, manifest tamper trip that is not operator-caused, atomic-gate fired in a copy-mode run.
>
> If a Sev1 happens, stop dogfood, capture `make debug-bundle RUN=<run-id> USB=/Volumes/<USB>` from project root, and start a new Claude session with the tarball attached.
